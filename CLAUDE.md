# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## 全局协作规则

### 沟通风格
- 默认用中文沟通，除非任务明确要求英文
- 先讲结论，再讲关键原因
- 复杂任务先给简短计划，再开始执行
- 不确定时说明假设和风险，不要装作确定

### 工作方式
- 动手前先阅读相关文件和已有约定
- 修改范围要尽量小，不做与任务无关的重构
- 遇到用户已经修改过的文件，先理解现状再继续
- 发现需求含糊时，先提出最关键的问题
- 能直接验证的结果，优先用命令或测试验证

### 安全边界
- 不要删除用户数据、历史记录、配置文件和密钥
- 不要自动执行 `git push`、部署、发布和破坏性迁移
- 需要大范围改动时，先说明影响范围
- 命令失败时不要硬猜，先读错误信息

### 交付标准
- 说明改了什么
- 说明验证了什么
- 测试不能跑时说明原因
- 标出仍然存在的风险和下一步建议

### 长期记忆维护
- 反复出现的偏好，沉淀到本文件
- 只在单个项目生效的规则，写到项目 CLAUDE.md
- 具体业务规范和长流程，放到项目 `docs` 目录

## LLM 编码行为准则 (Andrej Karpathy)

**权衡**: 这些准则偏向谨慎而非速度。对于简单任务可灵活处理。

### 1. 先思考再编码

**不要假设。不要掩盖困惑。明确表达权衡。**

实现之前:
- 明确陈述假设。如果不确定，提问
- 如果存在多种解释，呈现出来 — 不要默默选一种
- 如果有更简单的方法，说出来。有理由时敢于反驳
- 如果有不清楚的地方，停下来。指出困惑点，提问

### 2. 简单至上

**解决问题的最少代码。不做多余的事。**

- 不添加超出需求的功能
- 不为一次性代码创建抽象
- 不添加未要求的 "灵活性" 或 "可配置性"
- 不处理不可能发生的错误场景
- 如果写了 200 行但 50 行就够，重写

问自己: "一个资深工程师会觉得这过于复杂吗？" 如果是，简化。

### 3. 外科手术式修改

**只动必须动的。只清理自己弄乱的。**

编辑现有代码时:
- 不要 "顺便改进" 相邻代码、注释或格式
- 不要重构没有坏的东西
- 匹配现有风格，即使你更喜欢另一种写法
- 如果发现无关的死亡代码，提出来 — 不要直接删

当你的改动产生孤立代码时:
- 删除你的改动导致不再使用的导入/变量/函数
- 不要删除已有的死亡代码，除非明确要求

检验标准: 每行被改的代码都应直接追溯到用户的请求。

### 4. 目标驱动执行

**定义成功标准。循环直到验证通过。**

将任务转化为可验证的目标:
- "加验证" → "为无效输入写测试，然后让测试通过"
- "修 bug" → "写一个能复现的测试，然后让它通过"
- "重构 X" → "确保重构前后测试都通过"

多步骤任务先给简短计划:
```
1. [步骤] → 验证: [检查点]
2. [步骤] → 验证: [检查点]
3. [步骤] → 验证: [检查点]
```

---

**这些准则生效的标志**: diff 中不必要的改动更少、因过度复杂导致的重写更少、澄清问题在实现之前提出而非犯错之后。
